# 项目内多角色识别与批量生成设计

## 作用

第 10 阶段在 `ScriptProject` 项目空间内增加 `CharacterCandidate` 候选角色层。用户上传并解析剧本资料后，可以先识别项目内可能出现的多个角色，再由用户确认、编辑、忽略，最后选择一个或多个候选批量生成数字人。

当前实现为 evidence-driven 规则抽取 + reviewer 复核 + 用户确认，不是真实 LLM 角色抽取，不做模型训练，也不声称系统已经真实理解剧本人物关系。

## CharacterCandidate 字段

- `candidate_id`：候选角色业务 ID，例如 `candidate_001`
- `project_id`：所属剧本项目，候选只能在当前项目内使用
- `name`：候选角色名
- `aliases`：别名列表，当前用 JSON 字符串保存
- `identity_hint`：身份提示
- `personality_hint`：性格提示
- `relationship_hints`：关系线索列表，例如“阿奇和戴丽拉是情侣”
- `evidence`：识别证据摘要，限制在 500 字以内
- `source_document_ids`：来源 `parsed_document_id` 列表
- `confidence`：规则置信度，当前范围为 0 到 1
- `candidate_status`：`pending / confirmed / ignored / generated / rejected`
- `canonical_name`：规范化后的候选名，用于同项目去重
- `display_name`：前端显示名，可由用户编辑
- `candidate_type`：`person / organization / place / prop / clue / document_term / structure_term / noise / unknown`
- `source_types`：证据来源，例如 `dialogue_label / identity_field / relationship_sentence / current_character / repeated_subject`
- `evidence_spans`：短证据片段列表
- `dialogue_count`：作为对白标签出现的次数
- `mention_count`：相关证据命中次数
- `relationship_evidence`：干净关系短句列表
- `confidence_level`：`high / medium / low`
- `needs_human_review`：是否需要用户确认
- `reviewer_provider / reviewer_status / reviewer_reason`：LLM 或规则 reviewer 的复核信息

## evidence-driven 抽取策略

v0.18.2 后候选抽取服务会优先走 Script Intelligence Pipeline，只处理当前 `project_id` 下的解析文档，先清洗文本、分层为 `script_segments`，再切为 `script_chunks`，最后从证据片段聚合候选。

系统不再因为文本中出现一个短词就直接创建角色候选。候选必须来自证据：

- `角色名：台词`
- `姓名：xxx`
- `角色：xxx`
- `角色名：xxx`
- `你是xxx / 我是xxx / 你的角色是xxx`
- `A和B是情侣 / 朋友 / 宿敌 / 兄妹 / 父女 / 母女 / 同学 / 恋人`
- `A喜欢B / A暗恋B / A爱B`
- 多次作为行动主体出现，但只能作为弱证据，不能单独升为高置信

测试集只用于自动化验收，不会写进业务逻辑。候选名长度通常限制为 2 到 5 个中文字符，并通过 `candidate_type` 把章节结构词、剧本功能词、文档结构词、场景 / 道具 / 组织词和 OCR 噪声归为非人物或待确认。

## X的Y 归属规则

对 `X的声音 / X的照片 / X的房间 / X的日记 / X的回忆` 等结构，系统使用通用结构规则处理：

- 如果 `X` 已是高置信人物候选，不创建 `X的Y` 新角色，而是把整句加入 `X` 的 `evidence_spans`。
- 如果 `X` 还不是高置信人物，只记录为低置信 observation，需要更多证据或用户确认。
- 这条规则不写死任何样本人名。

## LLM reviewer 复核

有完整 `openai_compatible` 配置时，`CandidateLLMReviewer` 会把候选摘要、证据片段、source_types、dialogue_count、mention_count、segment_type 摘要和关系证据交给模型复核，并要求返回严格 JSON。

没有 API Key、模型调用失败或 JSON 解析失败时，自动回退 `RuleCandidateReviewer`。Reviewer 不会接收完整剧本文本，只处理有限长度的证据摘要，避免把整本商业剧本无限制发送给外部模型。

## 关系线索来源

`relationship_hints` 只来自上传资料里的规则句式和用户后续编辑，不代表模型真实理解剧情。如果资料里只有“情侣”这类手填提示，系统只能把它视为 `relationship_hint` 或候选备注，而不是剧本事实。

## 去重策略

重复调用 `POST /api/projects/{project_id}/candidates/extract` 时，同一项目内同名候选不会重复创建。

- 如果候选状态是 `pending / confirmed / ignored`，会更新证据、关系线索、来源文档和置信度
- 如果候选状态是 `generated`，默认不再覆盖，避免已生成角色被抽取结果扰动
- 不同项目内即使角色同名，也会创建各自项目内的候选，保持资料隔离

## 状态流转

- `pending`：规则识别后待用户确认
- `confirmed`：用户编辑或确认后的候选
- `ignored`：用户忽略，不物理删除
- `generated`：已由批量生成接口生成数字人

本阶段不做“重新生成已 generated 候选”。如果再次提交已生成候选，接口会返回 `skipped`。

## 批量生成流程

`POST /api/projects/{project_id}/characters/batch-generate` 接收候选 ID 列表、用户扮演身份、默认关系提示和可选关系覆盖。

每个候选都会复用现有单角色生成主流程：

1. 校验候选属于当前项目
2. 跳过 `generated / ignored` 候选
3. 自动选取当前项目上传资料
4. 调用现有 `generate_character`
5. 生成 `Character`
6. 生成 `CharacterProfile`
7. 构建 `KnowledgeChunk`
8. 生成 Q 版桌宠
9. 创建默认 `CharacterSettings`
10. 将候选状态标记为 `generated`

关系提示优先级：

1. `relationship_overrides[candidate_id]`
2. 与该候选相关的 `CharacterRelationship` 短摘要
3. 候选 `relationship_hints` 的第一条
4. `default_relationship_hint`

批量生成限制：

- `candidate_type != person` 的候选默认跳过
- `confidence_level=medium/low` 且未被用户 `confirmed` 的候选默认跳过
- `ignored / rejected / generated` 候选默认跳过

## project_id 隔离

候选抽取和批量生成都严格限定在 `project_id` 内：

- 候选只能从当前项目的解析文档中生成
- 批量生成不会使用其他项目的候选
- 批量生成不会使用其他项目的上传资料
- 生成出的角色、档案和知识片段继续绑定当前项目
- 后续聊天检索仍按项目隔离，避免同名角色跨剧本串库

## 前端流程

项目详情页按区域展示：

1. 项目摘要
2. 项目资料上传和解析
3. 高置信角色候选
4. 待确认候选
5. 非人物 / 噪声候选
6. 批量生成数字人
7. 已生成数字人列表

用户可以先修正候选的身份提示和关系线索，再批量生成。

## 后续升级

后续可以把当前规则抽取替换为真实 LLM 角色抽取，输出更完整的身份、阵营、关系网络、隐藏信息和多角色可玩性分析。升级时仍应保留 `CharacterCandidate` 作为用户确认层，避免模型误识别直接生成错误角色。

## v0.18.2-fixed 误识别整改

本次整改把候选晋级规则从“关系句命中即可高置信”收紧为“强证据 + reviewer 完成 + 用户可确认”的组合判断：

- `relationship_sentence` 只能作为关系证据或中低置信 observation，不能单独把候选提升为 `high person`。
- 高置信人物必须来自 `current_character / filename / title / identity_field / llm_review`，或来自多次 / 组合对白标签证据。
- `personhood_validator` 会把代词短语、抽象概念、章节结构词、线索/真相/玩家/主持人、`X的Y` 结构、从句碎片等降级为非人物或待确认。
- reviewer 必须同步完成；真实 LLM 不可用时写入 `reviewer_provider=rule`、`reviewer_status=rule_fallback`，不让前端长期显示 pending。
- 批量生成默认只允许高置信且不需要人工复核的 `person`，或用户手动 `confirmed` 的候选。
- 前端“待确认候选”和“非人物 / 噪声候选”默认折叠，低/中置信候选必须先确认为角色。
# v0.18.3-rc1 补充：LLM-first 候选复核

角色候选识别现在优先进入 LLM-first Script Understanding 链路。规则抽取负责证据收集、噪声分类和 fallback；有真实 `openai_compatible` API Key 时，候选会交给结构化 LLM 复核，输出 `candidate_type`、`confidence_level`、`needs_human_review`、`llm_reason` 和 evidence 摘要。

测试集只用于自动化验收，不写入业务逻辑。复杂剧本仍需要用户在前端确认候选后再批量生成。
