# 项目开发说明

## 1. 当前阶段目标

第 0 阶段已经完成可运行 MVP 工程骨架。第 1 阶段在保留原主流程的基础上，新增真实文件解析与文本抽取：

1. 打开小程序
2. 查看数字人聊天列表
3. 点击加号进入创建页
4. 上传剧本资料
5. 解析 TXT / DOCX / PDF 文本
6. 展示解析状态、摘要和字数
7. mock 生成数字人角色
8. mock 生成 Q版桌宠
9. 进入聊天页并继续原有陪伴流程

当前第 5.1 阶段仍是工程骨架整改，不代表系统已经能真实理解剧本。人物关系只能来自三类来源：

- 用户在创建数字人时手动填写的目标角色、用户身份和关系提示
- 上传文件解析文本中被规则关键词命中的信息
- 开发自测时临时构造的 sample TXT 测试内容

如果没有上传正式剧本，系统不应声称已知道真实人物关系；如果只填写了“情侣”，它只是 `relationship_hint`，不是剧本事实。

## 2. 模块划分

### 前端

- `pages/`：页面
- `components/`：通用组件
- `api/`：后端接口访问封装

### 后端

- `api/`：REST 路由
- `models/`：SQLAlchemy 数据模型
- `schemas/`：Pydantic 请求 / 响应模型
- `services/`：上传、解析、mock 业务逻辑
- `db/`：数据库初始化、会话管理
- `uploads/`：上传文件本地存储目录
- `static/mock/`：mock 头像和桌宠图片

## 3. 当前核心数据

角色结构已预留以下字段：

- `character_name`
- `character_identity`
- `user_persona_name`
- `relationship_with_user`
- `relationship_stage`
- `intimacy_level`

当前还包含：

- 角色头像
- 角色说明
- 性格
- 最近消息
- Q版桌宠信息
- 分层记忆
- 上传文件记录
- 解析文档记录

## 4. 后续扩展方向

1. OCR 服务接入与图片文本提取
2. RAG、角色抽取、关系抽取
3. 真实模型生成服务
4. 聊天会话与长期记忆
5. 用户体系与多角色管理
6. 桌宠资产生成、动作系统、资源管理
7. 测试、日志、异常处理、部署

真实模型调用也受环境变量控制：未配置 `LLM_API_KEY` 时不会调用真实模型，只使用 `MockLLMProvider` 生成规则测试回复；配置完整 provider、key、base_url 和 model 后才会尝试调用，失败时仍会回退 mock。
