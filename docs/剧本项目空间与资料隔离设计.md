# 剧本项目空间与资料隔离设计

## 1. 为什么需要 ScriptProject

剧本杀角色数字人会同时处理多个剧本资料包。不同剧本里可能出现同名角色，例如都叫“阿奇”，但人物关系、秘密、剧情阶段完全不同。

如果系统只按角色名检索知识库，就会出现跨剧本串资料、串角色、串知识片段的问题。这会直接破坏角色一致性，也会让防剧透策略失效。

因此第 9 阶段新增 `ScriptProject`，把上传资料、解析文档、角色、角色设定档案、知识片段、聊天会话、聊天消息、长期记忆和调教配置都绑定到同一个 `project_id`。

## 2. ScriptProject 字段

- `id`：数据库自增 ID。
- `project_id`：业务 ID，例如 `project_001`。
- `title`：剧本项目名称。
- `description`：项目简介。
- `source_type`：资料来源，当前支持 `upload / manual / demo`。
- `cover_image`：封面图，可为空。
- `project_status`：项目状态，当前支持 `active / archived`。
- `created_at`：创建时间。
- `updated_at`：更新时间。

## 3. project_id 贯穿链路

以下表新增 nullable `project_id`：

- `uploaded_files`
- `parsed_documents`
- `characters`
- `character_profiles`
- `knowledge_chunks`
- `chat_sessions`
- `chat_messages`
- `memories`
- `character_settings`

`project_id` 可为空是为了兼容旧数据和旧流程。旧数据不绑定项目时仍可继续使用原有逻辑。

新项目流程中：

1. 创建 `ScriptProject`。
2. 在项目详情页上传资料，`UploadedFile.project_id` 写入项目 ID。
3. 文件解析时，`ParsedDocument.project_id` 继承上传文件。
4. 项目内生成角色时，`Character.project_id` 写入项目 ID。
5. 角色设定档案、知识片段、聊天会话、聊天消息、长期记忆、角色调教配置都继承角色项目 ID。

## 4. 防止跨项目混用资料

`POST /api/characters/generate` 支持可选 `project_id`。

如果请求带 `project_id`，后端会检查 `uploaded_file_ids` 对应文件是否全部属于同一项目。不一致时直接拒绝，避免项目 A 的资料被拿去生成项目 B 的角色。

`POST /api/knowledge/build` 同样会检查 `parsed_document.project_id` 是否与请求项目一致。

## 5. 知识检索项目隔离

聊天检索时：

1. 如果 `character.project_id` 存在，只检索 `project_id == character.project_id` 的 `knowledge_chunks`。
2. 在项目范围内优先检索 `character_id == 当前角色` 的 chunks。
3. 如果没有角色绑定 chunks，才允许在同一个项目内按 `target_character_name` 兜底。
4. 兜底检索也不能跨项目。
5. 如果 `character.project_id` 为空，使用旧逻辑兼容，`retrieval_project_scope=legacy_no_project`。

`prompt_debug` 增加：

- `project_id`
- `retrieval_project_scope`

可选值：

- `project_bound`：当前检索被限制在角色所属项目内。
- `legacy_no_project`：旧数据无项目，使用兼容逻辑。
- `none`：未命中可用项目范围。

## 6. 同名角色跨剧本风险

示例：

- 项目 A：《流浪叙事》里，阿奇和戴丽拉是情侣。
- 项目 B：《另一个剧本》里，阿奇和小雪是宿敌。

如果只按角色名“阿奇”检索，项目 A 的阿奇可能命中项目 B 的“宿敌”资料。第 9 阶段通过 `project_id` 限定检索范围，避免这种串库风险。

## 7. 前端项目页流程

新增页面：

- `pages/projects/index`：项目列表与创建项目。
- `pages/project-detail/index`：项目 summary、项目内上传资料、项目内创建角色、项目角色列表。

首页保留旧数字人列表，同时新增“剧本项目”入口。旧流程不传 `project_id` 仍然可用。

## 8. 后续扩展

后续可以在同一个 `ScriptProject` 中支持一个剧本自动抽取多个角色、多角色共享同一个资料包、项目级别剧透进度和章节解锁。
