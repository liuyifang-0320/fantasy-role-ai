# 对话引擎设计

## 1. Chat Engine 的作用

`Chat Engine` 是第 3 阶段新增的对话编排层，用来把一次聊天请求拆成稳定的工程步骤：

1. 读取角色与角色设定档案
2. 读取最近上下文
3. 组装 Prompt
4. 调用统一的 Provider
5. 保存聊天记录
6. 返回角色化回复、桌宠动作和调试信息

当前阶段仍不接真实 LLM，重点是先把未来真实模型需要的管道搭好。

## 2. PromptBuilder 的输入输出

输入：

- `character`
- `profile`
- `recent_messages`
- `user_message`

输出：

- `system_prompt`
- `context_prompt`
- `user_prompt`
- `context_message_count`
- `uses_character_profile`

## 3. Prompt 结构

### `system_prompt`

包含：

- 角色名
- 角色身份
- 性格
- 说话风格
- 用户扮演身份
- 关系设定
- 当前剧情阶段
- 已知事实
- 隐藏秘密
- 防剧透规则

### `context_prompt`

- 最近 6 条历史消息
- 按 `role: content` 结构拼接

### `user_prompt`

- 当前用户输入

这套结构后续可以直接替换为真实 LLM 的输入组装层。

## 4. MockLLMProvider 规则

当前 `MockLLMProvider` 只做轻量规则回应：

- `记得我吗 / 还记得我吗`：确认用户身份与关系
- `难过 / 累 / 想哭 / 不开心`：返回安抚回复，`pet_action=comfort`
- `秘密 / 真相 / 凶手`：触发防剧透，`pet_action=thinking`
- 其他输入：返回普通角色化陪伴回复

回复必须保留角色名和关系感，避免变成普通客服语气。

## 5. 聊天会话和消息如何落库

### `chat_sessions`

- 保存一段对话的会话级信息
- 包含 `session_id`、`character_id`、`title`、创建时间、更新时间

### `chat_messages`

- 保存每条 user / assistant 消息
- 记录 `pet_action`、`memory_hint`
- 记录本轮生成使用的 `prompt_snapshot`
- 当前 `prompt_snapshot` 最多保存前 2000 字，便于调试

## 6. 后续如何替换为真实 LLM Provider

后续只需要在 Provider 层新增真实实现，例如：

- `OpenAIProvider`
- 其他模型 Provider

`PromptBuilder`、会话落库、前端协议都不需要推翻，只需要把 `MockLLMProvider` 替换为真实 Provider。

## 7. prompt_debug 上线前如何关闭

当前 `prompt_debug` 会返回：

- `system_prompt_preview`
- `context_message_count`
- `uses_character_profile`

上线前可通过以下方式关闭：

1. 在接口层增加配置开关
2. 关闭 `prompt_debug` 字段返回
3. 保留数据库里的 `prompt_snapshot` 仅供内部调试
