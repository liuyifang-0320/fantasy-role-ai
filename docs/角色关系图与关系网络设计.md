# 角色关系图与关系网络设计

## 为什么需要角色关系图

剧本杀角色数字人不只需要单个角色设定，还需要知道项目内角色之间的关系。关系图用于帮助系统保持关系一致性，避免同名角色、跨剧本资料、临时用户说法对角色关系造成污染。

第 11 阶段仍然是规则抽取 + 用户可编辑，不是真实 LLM 关系抽取，不做复杂图数据库。v0.18.2 后，关系抽取会优先使用 Script Intelligence Pipeline 清洗后的文本和文档分层结果，减少 OCR 噪声、页眉页脚、章节词对关系识别的干扰。

## CharacterRelationship 字段

- `relationship_id`：关系业务 ID，例如 `rel_001`
- `project_id`：所属剧本项目
- `source_character_name`：关系主体，例如“阿奇”
- `target_character_name`：关系对象，例如“戴丽拉”
- `relation_type`：关系类型，例如情侣、朋友、宿敌、喜欢、暗恋、旧识
- `relation_summary`：关系摘要，例如“阿奇和戴丽拉是情侣”
- `evidence`：证据片段，不超过 500 字
- `source_document_ids`：来源 `parsed_document_id` 列表
- `confidence`：规则置信度，当前范围为 0 到 1
- `relationship_status`：`pending / confirmed / ignored`
- `created_at / updated_at`：创建和更新时间

## 当前规则抽取策略

关系抽取服务扫描当前项目的 `ParsedDocument.raw_text`，识别以下句式：

- `A和B是情侣 / 朋友 / 宿敌 / 兄妹 / 父女 / 母女 / 同学 / 恋人`
- `A与B是情侣 / 朋友 / 宿敌...`
- `A喜欢B`
- `A爱B`
- `A暗恋B`
- `A认识B`
- `A曾经和B分别 / 重逢`
- `A是B的父亲 / 母亲 / 女儿 / 儿子 / 朋友 / 恋人`

候选名限制为 2 到 5 个中文字符，并通过通用噪声分类过滤章节结构词、剧本功能词、文档结构词、场景 / 道具 / 组织词和 OCR 噪声。测试样本中的人名不会写进业务规则。

`relationship_hints` 只保存干净短句，例如“阿奇和戴丽拉是情侣”“小雪和阿奇是宿敌”“戴丽拉喜欢阿奇”；上下文片段进入 `evidence`，避免把整段剧本文本塞进候选关系线索。

## 关系去重策略

同一项目内，如果 `source_character_name + target_character_name + relation_type` 已存在，不重复创建，只更新摘要、证据、来源文档和置信度。

重复调用抽取接口不会重复创建同关系。

## 正反向关系合并策略

对对称关系做正反向合并，例如：

- 阿奇和戴丽拉是情侣
- 戴丽拉和阿奇是情侣

会视为同一条关系。当前对称关系包括情侣、恋人、朋友、宿敌、兄妹、父女、母女、同学、旧识、分别、重逢、认识。

喜欢、暗恋、父亲、母亲、女儿、儿子等方向性较强的关系保留方向。

## project_id 隔离

所有关系都绑定 `project_id`。

- 抽取只扫描当前项目解析文档
- 列表、详情、图接口只返回当前项目关系
- 批量生成角色只读取当前项目内相关关系
- 聊天 Prompt 只注入当前角色所属项目内的关系

如果不同项目都有同名“阿奇”，也不会共享关系图。

## 注入 PromptBuilder

聊天时 Chat Engine 会读取当前角色相关的 `confirmed / pending` 关系，传入 PromptBuilder。

PromptBuilder 会生成 `relationship_prompt`，例如：

```text
项目角色关系图：
- 阿奇 与 戴丽拉：情侣。阿奇和戴丽拉是情侣
- 阿奇 与 小雪：宿敌。小雪和阿奇是宿敌
```

`system_prompt` 增加规则：

- 可以参考项目角色关系图维持关系一致性
- 如果关系图和用户临时说法冲突，优先遵循项目资料和角色设定
- 不要主动暴露隐藏关系，除非剧本资料允许

`prompt_debug` 增加：

- `active_relationship_count`
- `relationships_preview`

## 剧透边界

关系证据如果来自 `truth_reveal / final_mission / hidden_secret` 等 heavy spoiler segment，会标记为 hidden / restricted。默认 `spoiler_mode=non_spoiler` 时，这些关系不会进入普通聊天可自由使用区域；只有用户或角色配置切换到更高剧透模式时，才允许更宽松地使用，同时仍受内容安全和防剧透规则约束。

## 影响批量角色生成

批量生成某个候选角色时，关系提示优先级为：

1. `relationship_overrides`
2. 与该候选相关的 `CharacterRelationship` 摘要
3. 候选 `relationship_hints`
4. `default_relationship_hint`

关系提示会限制在 200 字内，避免把长证据塞进角色生成流程。

## 前端关系图展示

项目详情页增加“角色关系图”区域：

- 抽取角色关系
- 展示关系列表
- 编辑关系摘要、主体、对象和类型
- 忽略关系
- 用线性卡片展示简易图：`阿奇 — 情侣 — 戴丽拉`

本阶段不引入复杂图形库。

## 后续升级

后续可以升级为真实 LLM 关系抽取，补充关系强度、阵营、显性/隐藏关系、时间线关系和剧透等级。更复杂项目可再升级到图数据库或图谱检索，但当前阶段先保留 SQLite 表结构和可编辑确认流程。
# v0.18.3-rc1 补充：LLM-first 关系抽取

关系抽取新增 LLM-first 复核骨架：规则层先抽取干净短句和证据摘要，真实 LLM 可用时再判断 `is_explicit / is_inferred`、`confidence_level`、`spoiler_level`、`visibility` 和是否需要人工确认。关系短句仍要求保持简洁，长上下文只能进入 evidence，不应塞入 relationship_hints。

hidden / heavy 关系默认不会进入 non_spoiler 普通聊天上下文。
