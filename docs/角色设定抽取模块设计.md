# 角色设定抽取模块设计

## 1. CharacterProfile 的作用

`CharacterProfile` 是角色生成前的一层结构化中间结果，用来把解析文档里的文本信息整理成后续模块可复用的数据。它不会替代角色本身，而是为真实 AI 能力做准备。

当前阶段它承担 4 个职责：

- 把上传资料中的角色信息沉淀为稳定字段
- 为后续真实 LLM 角色 Prompt 提供结构化输入
- 为防剧透与信息边界控制提供基础规则
- 为恋陪关系设定、剧情阶段和记忆系统预留接口

## 2. 当前规则抽取策略

当前版本不接真实 LLM，使用规则抽取加 mock 补全：

1. 文本中出现 `target_character_name`，视为目标角色存在。
2. 文本中出现 `user_persona_name`，视为关系存在上下文。
3. 检测 `情侣 / 恋人 / 喜欢 / 爱 / 分别 / 重逢 / 秘密 / 凶手 / 神社 / 旧街区` 等关键词。
4. 依据关键词生成：
   - 角色身份
   - 性格
   - 说话风格
   - 剧情阶段
   - 关系摘要
5. 从文本摘要中拆分最多 3 条 `known_facts`。

当文本里能命中角色名、用户身份或关键词时，`extraction_status` 为 `success`；否则进入 fallback mock。

需要注意的是，当前阶段不能声称系统已经真实理解剧本人物关系。关系来源只包括用户手填的 `target_character_name`、`user_persona_name`、`relationship_hint`，上传文本中被规则命中的关键词，以及开发自测时临时构造的 sample TXT。若用户没有上传正式剧本，生成出的关系摘要只是测试或占位结果；若用户只填写“情侣”，系统只能把它当作关系提示，不应当作剧本事实。

## 3. fallback mock 策略

当资料不足以稳定抽取时，使用以下保底字段：

- `extracted_identity`：由剧本资料生成的关键角色
- `extracted_personality`：温柔、敏感、带有故事感
- `speaking_style`：克制、暧昧、带有回忆感
- `background_summary`：根据上传资料生成的角色背景摘要，当前为规则抽取版本
- `relationship_summary`：根据 `relationship_hint` 生成
- `story_stage`：初识升温
- `known_facts`：从文本摘要中截取 3 条，若无文本则返回占位说明
- `hidden_secrets`：当前无法确认，默认不主动暴露剧本核心秘密
- `spoiler_guardrails`：不得主动剧透；不得编造未上传资料中的关键真相

fallback mock 只用于保持流程可用，不代表模型理解了未上传资料，也不会凭空证明人物关系。

## 4. 如何为后续 LLM Prompt 服务

后续接入真实 LLM 时，可以把 `CharacterProfile` 直接作为 Prompt 组装层输入：

- `extracted_identity` 决定角色定位
- `extracted_personality` 与 `speaking_style` 决定表达风格
- `background_summary` 提供背景上下文
- `relationship_summary` 与 `story_stage` 约束角色和用户当前关系
- `known_facts`、`hidden_secrets`、`spoiler_guardrails` 共同定义可说与不可说的边界

这样能把“角色生成”和“资料解析”拆开，后续替换真实模型时不需要推翻主流程。

## 5. 如何为防剧透服务

`hidden_secrets` 与 `spoiler_guardrails` 是后续剧透控制的核心字段：

- `hidden_secrets` 表示角色默认不应主动暴露的信息
- `spoiler_guardrails` 表示生成回复时必须遵守的规则
- 当后续接入 RAG 或真实 LLM 时，这两类字段可以一起进入系统提示词或检索过滤逻辑

## 6. 如何为恋陪关系设定服务

恋陪体验不只依赖一句关系标签，还需要上下文：

- `relationship_summary` 说明双方当前关系
- `story_stage` 说明剧情推进位置
- `speaking_style` 决定情感表达方式

后续关系阶段演进、长期记忆、主动问候、暧昧程度控制，都可以基于这些字段继续扩展。
