# 用户体系与数据隔离设计

## 1. 设计目标

第 13 阶段新增用户体系骨架，目标不是正式登录，而是先让项目内所有用户数据具备明确归属。后续接入微信登录、手机号登录或其他账号体系时，可以沿用 `user_id` 贯穿链路，避免不同用户之间看到彼此的剧本项目、角色、聊天、记忆和桌宠资产。

当前阶段仍然是开发测试形态：

- 不做正式微信登录。
- 不做手机号登录。
- 不存储真实敏感身份信息。
- 使用 `dev_mock` 用户机制模拟 current user。

## 2. User 数据模型

`users` 表用于保存开发测试用户和未来正式用户的基础信息。

字段说明：

- `id`：数据库自增主键。
- `user_id`：业务 ID，例如 `user_001`、`dev_user`。
- `nickname`：用户昵称。
- `avatar`：用户头像地址，可为空。
- `auth_provider`：认证来源，当前支持 `dev_mock`，预留 `wechat`、`phone`。
- `openid`：微信 openid，当前为空，后续预留。
- `unionid`：微信 unionid，当前为空，后续预留。
- `phone`：手机号，当前为空，后续预留。
- `user_status`：用户状态，`active / disabled`。
- `created_at / updated_at`：创建与更新时间。

## 3. dev_mock 登录机制

后端通过 `X-User-Id` 请求头识别当前用户：

```http
X-User-Id: user_001
```

规则：

1. 如果请求头带 `X-User-Id`，后端会使用该 user_id 获取当前用户。
2. 如果数据库中不存在该用户，开发阶段会自动创建一个 `dev_mock` 用户。
3. 如果请求头缺失，默认使用：

```json
{
  "user_id": "dev_user",
  "nickname": "开发测试用户",
  "auth_provider": "dev_mock"
}
```

这只是开发环境 mock 登录，不代表正式账号系统。前端用户中心页会把 `user_id` 存到本地 storage，并在每个请求里自动携带 `X-User-Id`。

## 4. user_id 贯穿链路

第 13 阶段已给以下核心数据增加 nullable `user_id`：

- `ScriptProject`
- `UploadedFile`
- `ParsedDocument`
- `Character`
- `CharacterProfile`
- `KnowledgeChunk`
- `CharacterRelationship`
- `CharacterCandidate`
- `ChatSession`
- `ChatMessage`
- `Memory`
- `CharacterSettings`
- `Pet`
- `PetAsset`

新创建的数据会写入当前用户的 `user_id`：

创建项目 -> 上传文件 -> 文件解析 -> 生成角色 -> 生成 CharacterProfile -> 构建 KnowledgeChunk -> 创建 Pet / PetAsset -> 聊天会话与消息 -> 自动记忆 -> 角色调教配置

这样即使两个用户拥有同名项目、同名角色、同名关系，也不会互相查询到对方的数据。

## 5. 权限校验策略

后端新增 access control 工具函数，对项目、文件、角色、记忆、候选、关系、PetAsset 等资源做统一校验。

规则：

1. `resource.user_id == current_user.user_id`：允许访问。
2. `resource.user_id is null`：开发阶段允许访问，作为旧数据兼容。
3. 其他情况：返回 `403 Forbidden`。

`user_id` 隔离优先于 `project_id` 隔离。也就是说，即使用户知道了别人的 `project_id` 或 `character_id`，只要 `user_id` 不匹配，也不能访问。

## 6. legacy_null_user 兼容

第 13 阶段之前的数据没有 `user_id`，因此新字段允许为空。开发阶段为了不破坏已有自测数据，`user_id is null` 的旧数据仍允许访问。

上线前必须处理：

- 将旧数据迁移到明确用户。
- 或关闭 `legacy_null_user` 访问。
- 不允许生产环境中长期保留“无主数据可被任意开发用户访问”的策略。

## 7. API 概览

新增认证相关接口：

- `GET /api/auth/me`：获取当前用户。
- `POST /api/auth/dev-login`：创建或返回开发 mock 用户。
- `POST /api/auth/switch-dev-user`：开发阶段切换 mock 用户。

所有业务接口都支持 `X-User-Id` 请求头。重点接口会按当前用户过滤或校验：

- 项目列表只返回当前用户项目。
- 文件上传、解析只允许当前用户操作自己的文件。
- 角色、聊天、记忆、角色调教、桌宠资产只允许当前用户访问。
- 候选角色和角色关系必须位于当前用户可访问的项目下。
- 知识库检索不能跨用户查询。

## 8. 前端用户中心

新增 `pages/user/index` 用户中心页：

- 展示当前 `user_id`、昵称、`auth_provider`。
- 支持输入昵称创建 dev mock 用户。
- 支持按 `user_id` 切换测试用户。
- 支持切回 `dev_user`。
- 明确提示当前不是正式登录，不要输入真实敏感信息。

首页和设置页也会展示当前用户状态。

## 9. 后续接入正式登录

后续接微信登录时，可以将登录流程替换为：

1. 前端调用微信登录能力拿到 code。
2. 后端换取 openid / unionid。
3. 后端查找或创建 User。
4. 后端签发正式登录态。
5. 前端不再手工传 dev user，而是由正式登录态确定 current user。

当前阶段只完成数据归属和隔离骨架，不实现正式登录。
