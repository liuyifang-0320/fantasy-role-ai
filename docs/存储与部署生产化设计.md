# 存储与部署生产化设计

## 1. 设计目标

第 14 阶段的目标不是直接上线，而是把本地开发型工程整理成可部署的配置骨架：

- 本地默认仍然使用 SQLite 和 local uploads。
- 生产环境可通过 `DATABASE_URL` 切换数据库。
- 文件上传通过 StorageProvider 抽象层保存。
- 文件记录具备 `storage_key / public_url / content_type / file_size` 等生产化字段。
- CORS、日志、错误处理和健康检查具备基础生产配置入口。

## 2. 为什么本地 uploads 不适合上线

`backend/app/uploads` 适合本地开发，但生产环境存在明显限制：

- 多实例部署时，每个实例本地文件不共享。
- 容器重启或重新部署可能导致本地文件丢失。
- 小程序 / H5 通常需要 HTTPS 可访问的公开 URL。
- 文件访问、备份、权限和 CDN 加速需要对象存储承担。

因此第 14 阶段引入 StorageProvider，让未来可以接入阿里云 OSS、腾讯云 COS、七牛、S3-compatible 服务等。

## 3. StorageProvider 设计

统一接口：

```python
save_upload(file, filename, content_type) -> StorageResult
get_public_url(storage_key) -> str
delete_file(storage_key) -> bool
```

统一返回：

- `storage_provider`：实际保存 provider，例如 `local`。
- `storage_key`：存储系统内的文件 key。
- `file_path`：兼容旧接口的文件路径。
- `public_url`：前端可访问 URL。
- `local_path`：本地路径，本地模式用于解析文件。
- `content_type`：上传文件 MIME 类型。
- `file_size`：文件大小，单位 byte。

## 4. LocalStorageProvider

默认启用：

- 文件保存到 `backend/app/uploads/`
- `public_url` 为 `/uploads/{storage_key}`
- 不依赖任何外部云服务
- 保留当前本地开发流程

## 5. S3CompatibleStorageProvider 骨架

预留给 S3-compatible 对象存储，读取环境变量：

- `STORAGE_PROVIDER=s3_compatible`
- `STORAGE_ENDPOINT_URL`
- `STORAGE_BUCKET`
- `STORAGE_ACCESS_KEY_ID`
- `STORAGE_SECRET_ACCESS_KEY`
- `STORAGE_REGION`
- `STORAGE_PUBLIC_BASE_URL`

当前阶段不要求真实上传。如果配置不完整或没有安装可选 SDK，会自动 fallback 到 local，不影响后端启动。

如果后续要真实启用，需要：

1. 安装对应 SDK，例如 `boto3` 或厂商兼容 SDK。
2. 在 provider 中实现真实上传、删除和 public URL 生成。
3. 配置 HTTPS 公开域名或 CDN。

## 6. CustomStorageProvider 骨架

预留给非 S3 协议的厂商：

- 阿里云 OSS
- 腾讯云 COS
- 七牛云
- 其他国内对象存储

当前阶段只保留接口和 fallback local。后续按具体厂商 SDK 扩展。

## 7. storage_key / public_url / file_path 区别

- `storage_key`：存储系统内的唯一 key，例如 `file_001_script.txt`。
- `public_url`：前端可访问地址，本地为 `/uploads/file_001_script.txt`，生产可为 CDN URL。
- `file_path`：兼容旧字段，当前本地模式等同于 `public_url`。

解析服务当前仍从本地文件读取。未来如果文件保存在云端，需要先下载到临时目录再进入 TXT / DOCX / PDF / OCR 解析链路。

## 8. 数据库配置

新增：

- `DATABASE_URL`
- `DATABASE_ECHO`

本地默认：

```env
DATABASE_URL=sqlite:///./fantasy_role_ai.db
DATABASE_ECHO=false
```

生产建议 PostgreSQL 或 MySQL。项目当前不强制安装对应驱动，需要部署方根据实际数据库安装：

- PostgreSQL 可选择 `psycopg` / `psycopg2`
- MySQL 可选择 `pymysql` / `mysqlclient`

## 9. CORS 配置

新增：

```env
CORS_ALLOW_ORIGINS=http://127.0.0.1:5173,http://localhost:5173
```

上线时应替换为真实 H5 域名、小程序后台配置域名或管理端域名。默认不在生产环境全开放。

## 10. 日志与错误处理

第 14 阶段新增轻量日志配置：

- 基础日志格式包含时间、级别和 logger 名称。
- 每个请求生成 `request_id`。
- 未捕获异常返回统一结构：

```json
{
  "detail": "Internal server error",
  "request_id": "..."
}
```

日志不会打印 API Key、云存储 Secret 等敏感配置。

## 11. 健康检查

保留：

- `GET /health`

新增：

- `GET /api/system/health`

返回数据库、存储、LLM、OCR、图片生成 Provider 的状态概览。该接口不会调用真实 LLM 或真实图片生成服务。

## 12. 上线前清单

- 配置正式 `DATABASE_URL` 并安装对应数据库驱动。
- 配置对象存储 Provider，确认上传、访问、删除策略。
- 配置 HTTPS 域名和 CDN。
- 配置 `CORS_ALLOW_ORIGINS` 为真实域名。
- 关闭或迁移 `legacy_null_user` 数据。
- 接入正式微信登录或其他登录体系。
- 将 `.env` 放在部署环境，不提交、不打包。
- 配置日志采集、错误监控和备份策略。
